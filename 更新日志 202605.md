### 1. 前端推流与子 Agent 会话污染

#### 用户反馈
- 子 Agent 的事件（user/assistant/tool 流式）会污染主对话区，例如切会话后原会话的子 Agent 用户气泡仍出现在主对话。
- 发送消息时切会话，消息会错发到另一个会话。
- 停止按钮响应卡顿，需等很久才能中断任务。
- 子 Agent 数量多时前端严重卡顿、滚动条不流畅。

#### 定位与分析
- SSE 事件处理中，`handleSubagentStreamEvent` 在 `currentSessionId !== runSessionId` 时直接返回 `false`，导致带 `agent_id` 的事件 **fall-through** 到普通 `renderEvent`，从而误渲染到主对话区域。
- 发送消息时 `sendMessage` 中多个 `await` 之间未锁定会话快照，导致消息发到切换后的会话。
- 停止按钮先 `await requestInterrupt` 再 `abort`，造成界面延迟。
- 子 Agent 卡片轮询（`runSubagentIncrementalSync`）与 SSE 并发，大量 HTTP 请求 + DOM 重绘导致主线程阻塞。
- 轮询中的 `finalizeLlmStreamChunks` 会提前结束当前流式块，造成流式 token 被截断成多个 feed 块。

#### 解决方案
- **P0 修复**
  - `handleSubagentStreamEvent` 跨会话时 **`return true`**（fail‑closed），阻止事件落到主对话。
  - SSE `ephemeral` 分支也优先判定 `agent_id` 并走子 Agent 通道，禁止 fall‑through。
  - `pauseCurrentRun` 改为先 `abort()` 再 fire‑and‑forget `requestInterrupt`。
  - `sendMessage` 顶部快照 `submitSessionIdInitial`，所有 `await` 后校验会话是否切换，若切换则不再渲染用户气泡。

- **P1 性能优化**
  - 父会话 SSE 运行时，子 Agent 轮询 **退避至 4–8s**，且 **并发降为 1**。
  - 轮询逻辑中去掉 `finalizeLlmStreamChunks`，避免打断流式块。
  - 卡片统计（`scheduleSubagentCardStats`）改为 **250ms 节流**，仅对可见卡片计算。
  - 溢出测量（`scheduleFeedChunkOverflowRefresh`）从 3 次 RAF 降为 1 次，streaming 中不 measure。

- **P2 体验优化**
  - 历史回放（`replayingMessages`）时禁止 upsert 子 Agent 卡片和按钮。
  - 删除子 Agent 接口与前端删除按钮（见第 4 条）。

**结果**：主对话区不再混入子 Agent 事件，停止/发送响应即时，多子 Agent 场景卡顿大幅缓解。

### 2. 配置向导重复跳转与 .env 重复行

#### 用户反馈
- 已经配置好 API Key，重启后仍跳转到配置向导。
- 每次保存向导会在 `.env` 末尾多出一行 `EXECUTOR_LLM=deepseek-v4-flash`。

#### 定位与分析
- **Bug①**：`_upsert_env_line` 只替换第一个匹配，不删除重复行，导致反复追加。
- **Bug②**：`_is_configured` 要求 7 项全非空，但 `_dotenv_last_assignments` 读取重复行时可能读到空值（例如被注释或 BOM 干扰），误判未配置。
- **根本原因**：`.env` 首行存在隐藏的 **UTF‑8 BOM**（`\ufeff`），正则匹配 key 失败，因此认为缺失。

#### 解决方案
- 增加 `_normalize_dotenv_key` 去除 key 前的 BOM。
- 修改 `_upsert_env_line` 替换所有匹配行（并删除多余行）。
- `_is_configured` 增加 fallback 逻辑：若最后一条值为空，则查找该 key 最后一个非空值。
- 新增 `/api/save_config` 统一走 `_apply_env_updates`，不再手工逐条 upsert。
- 提供脚本清理当前 `.env` 中的重复与 BOM。

**结果**：向导不再误跳，`.env` 干净无重复，BOM 不再影响解析。

### 3. 子 Agent 消息序列脏数据与分段流式碎片

#### 用户反馈
- 子 Agent 过程里出现大量空“信息”条目。
- 主 Agent 在 subagent 未完成时追问，导致子 Agent 的 reasoning/response 被切成多个“思考”块。
- 子 Agent 的完整回复被切成多个 feed‑item 连续出现。

#### 定位与分析
- 后端 `child_emit` 将很多低价值状态（空 status、`New Agent Loop Start` 等）也写入了 `ui_events.json`。
- 前端轮询时，全量重放事件会把已合并的流式 delta 再次当成普通事件渲染。
- 同时 SSE 转发的 delta 与轮询拉取的全量消息冲突，导致块被截断并重新开头。

#### 解决方案
- **后端过滤**：在 `agent_subagent.py` 增加 `_is_low_value_subagent_ui_event`，过滤空 status、loop 标记等，不再持久化。
- **前端合并重放**：`hydrateSubagentTurnProcess` 中，对 deferred 事件里的 `llm_reasoning_delta` / `llm_response_delta` 按流式合并处理（`appendLlmStreamDelta`），而不是逐条 `renderEvent`。
- **去重**：轮询增量同步时根据 `subagentCardEventCount[agentId]` 跳过已被 SSE 渲染过的事件。
- **禁用轮询中的 finalize**：SSE 在跑期间轮询只更新计数，不重绘 body。

**结果**：子 Agent 卡片内不再出现空“信息”，流式输出连续，不会出现碎片块。

### 5. 上下文压缩（Context Compression）全面改造

- 压缩时出现 `WARNING - note_key_context 无 <summary> 块`，随后前端报 `network error`。
- 压缩状态行百分比与右上角不一致，且经常出现“已经 12% 却还要压三轮 + 截尾”。
- 希望压缩流程更可控：明确进入/退出条件、轮次限制、微压逻辑统一、支持多轮摘要但避免无意义循环。
- 要求 **压缩前备份**，改写被压缩的 user 消息时能恢复上下文。
- 压缩执行器的 prompt 从纯文本改为消息序列（保留 role / tool 结构）。
- 将 `compress_old_history` / `note_key_context` 输入统一为同一段历史消息序列，且 key 摘要改为更新制而非追加。

#### 解决方案

##### 5.1 统一估算口径与退出条件
- 新增 `estimate_full_input_tokens_for_llm_history` 函数，与右上角计算完全一致。
- 压缩判断与达标判定都使用 **整包 token ≤ `CONTEXT_WINDOW × CONTEXT_COMPRESS_TARGET_RATIO`**（默认 0.5），不再依赖复杂的 `target_tokens` 公式。
- 移除 `_target_work_tokens` 及相关逻辑。

##### 5.2 微压（Phase D/E）统一截断规则
- 阶段 D：对 `ls / glob / grep / delete_file / web_search` 工具结果做首尾保留（长度 = `max(MICRO_SHRINK_TOOL_CHARS, MICRO_SHRINK_FAT_TOOL_FLOOR)`）。
- 阶段 E：对“完整保留×3 轮之前”的中间 assistant/tool 做微压，保留 user 与终稿 assistant。
- 统一微压截断为 **首尾各保留 k 字符，中间写 `...已微压省略xxx字符...`**，变量可调。

##### 5.3 并行摘要 + key 提炼 + 微压，顺序调整为 system → 对话 → 末条 user
- 重写 `_compress_executor_message_bundle`，顺序：
  `SystemMessage(compress_old_history 指令)` → 多条 user/assistant/tool 消息 → 末条 `UserMessage`（明确摘要任务 + key 摘录）。
- `note_key_context` 同理，末条 user 要求输出 `<analysis>` + `<summary>`。
- 二者输入 **同源**（`prefix_snap`），共用 `_compress_round_parallel` 并行执行，最终合并。

##### 5.4 压缩前备份与改写恢复
- `SessionManager.backup_llm_history_before_compress`：写入 `llm_history_<timestamp>.json`，并记录到 `metadata.llm_history_compress_backups`。

- 多轮压缩只备份一次（入口处），不再每轮备份。

##### 5.5 前端状态行改进
- 压缩状态行百分比改为 **整包百分比**（与右上角一致）。
- 微压提示改为 `【上下文裁剪】已对非关键信息进行裁剪` 等，摘要轮提示为 `【上下文摘要】完成 N 轮历史摘要；完成关键信息、经验与结论的记录`。

##### 5.6 key_context 更新制 & todo 独立

##### 5.7 新增环境变量
- `CONTEXT_COMPRESS_MAX_ROUNDS` 默认 3（最多几轮并行摘要）。
- `CONTEXT_COMPRESS_TARGET_RATIO` 默认 0.5（目标比例）。

### 7. Shell 工具鲁棒性提升（run_shell）

#### 用户需求与问题
- Windows 下无 Git Bash 时，`run_shell` 回退到 `cmd`，导致多行 `python -c` 被截断（`\n` 被当命令分隔符）。
- 多层引号嵌套失真（`\\"` 每经过一层 shell 就被重解析）。
- 二进制工具输出（如 ffmpeg）可能灌爆上下文。
- 希望学习 Claude Code 的做法：Bash 用 `eval` + `shlex.quote`，PowerShell 用 `-EncodedCommand`，并移除 cmd 回退。

#### 解决方案
- **Windows 上完全移除 `cmd` 回退**：无 Git Bash 时必须使用 PowerShell，找不到则报错。
- **Bash 路径**：用户命令包裹为 `eval {shlex.quote(user_cmd)}`，避免外层引号解析破坏脚本。
- **PowerShell 路径**：使用 `-EncodedCommand` + UTF‑16LE Base64，避免任何外层 shell 再次解释。
- **二进制输出检测**：增加 `_summarize_shell_stream_if_binary_like`，当 stdout/stderr 中空字节比例 >3% 或解码后替换字符/控制符比例 >12% 时，折叠为短摘要 + hex 预览。
- **环境变量**：`RUN_SHELL_MATERIALIZE_PYTHON_C` 控制是否自动将复杂的 `python -c` 落盘为临时 `.py` 执行并删除。
- **删除别名 `run_cli`**：统一工具名称为 `run_shell`，不再兼容旧名。
- **更新提示词与工具描述**：明确说明当前会话实际使用的 shell，并提醒避免 `wc`/`grep` 等假设。

**结果**：Windows 下多行 `python -c` 不再破碎，复杂引号可用性提升，二进制输出不会撑爆上下文，系统更鲁棒。

### 11. 会话元数据并发安全与原子写入

#### 问题
- `metadata.json` 因多线程/多协程并发写入导致文件损坏（空文件或截断 JSON），进而引发会话加载失败、崩溃等问题。

#### 定位与分析
- 多个地方直接调用 `_save_metadata` 写入文件，未使用任何锁机制。
- 写入时未采用原子操作，若写入中途被中断则留下不完整文件。

#### 解决方案
- **按会话粒度加锁**：`_session_metadata_lock(session_id)` 返回每个会话独立的 `threading.Lock`。
- **原子写入**：`_save_metadata_unlocked` 先将 JSON 写入临时文件（`{path}.tmp`），然后调用 `os.replace(tmp_path, path)`，保证原子替换。
- **容错读取**：增加 `_parse_metadata_json_raw` 函数，使用 `json.JSONDecoder` 的 `raw_decode` 尝试从损坏文件中提取第一个完整 JSON 对象；若失败则回退到备份或空字典。
- **强制重建索引**：在 `refresh_sessions_index_from_disk` 中增加检测，若发现 `.get("is_subagent")` 为 `True` 则跳过，避免误把 subagent 加入主会话索引。
- **所有涉及 metadata 写操作均改为通过锁保护 + 原子写入**，包括 `patch_subagent_metadata`、`append_pending_subagent_result`、`delete_session`、`set_session_archived`、`set_session_pinned`、`set_session_name`、`request_interrupt`、`clear_interrupt` 等。

**结果**：`metadata.json` 不再损坏，即使在并发写入或进程异常终止时也能保证文件完整，会话稳定性和数据安全显著提升。
